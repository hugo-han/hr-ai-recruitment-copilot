## 一、项目背景

公司当前招聘流程主要依赖 HR 人工完成。随着业务快速发展，招聘需求增加，HR 团队面临效率瓶颈，主要表现为：

1. JD（职位描述）编写耗时；
2. 简历筛选工作量大；
3. 候选人与岗位匹配依赖个人经验，标准不统一；
4. 面试评价缺乏统一标准，主观性强；
5. 招聘数据分散，无法有效沉淀与分析。

本项目旨在建设 **HR 招聘 AI 助手**，通过 AI 能力覆盖招聘全过程，降低 HR 重复劳动、提升筛选效率、提高招聘质量，并建立可复用的招聘数据资产。

**业务目标**：
- 降低 HR 重复工作；
- 提升候选人筛选效率；
- 提高招聘质量；
- 建立招聘数据资产。

**成功指标（验收基线）**：
- JD 生成时间减少 ≥ 80%；
- 简历初筛时间减少 ≥ 70%；
- 面试评价标准统一（可量化为评价指标模板覆盖率 ≥ 90%）；
- 招聘数据可分析（关键指标可视化覆盖 ≥ 4 项）。

## 二、用户角色

| 角色 | 描述 | 核心诉求 |
|---|---|---|
| HR 招聘专员 | 负责日常 JD 发布、简历筛选、候选人沟通 | 快速生成 JD、自动筛选简历、自动生成候选人分析 |
| HR 负责人 | 管理招聘团队与整体招聘策略 | 查看招聘效率指标、分析渠道效果、优化招聘策略 |

| 系统管理员 | 负责系统配置、权限与数据管理（补充角色） | 管理用户权限、岗位库、知识库与模型配置 |

## 三、使用场景

### 场景 1：HR 快速生成 JD
HR 招聘专员收到新招聘需求，输入岗位名称、岗位等级、业务要求，系统自动生成 JD、岗位画像与技能要求，HR 微调后发布。

### 场景 2：简历批量筛选
HR 招聘专员将一批候选人简历导入系统，系统针对目标岗位自动计算匹配评分，输出优势分析与风险提示，HR 按评分排序决定进入面试环节的候选人。

### 场景 3：面试辅助与评价
面试官面试后录入面试记录，系统自动生成面试总结、能力评价与推荐建议，面试官确认后归档。

### 场景 4：招聘数据分析
HR 负责人进入数据分析看板，查看招聘周期、转化率、渠道效果等指标，据此优化招聘策略。

## 七、验收标准（Acceptance Criteria）

### AC-US-01（JD 生成）
1. 输入岗位名称、等级、业务要求后，系统在 15s 内返回结果；
2. 输出包含 JD、岗位画像、技能要求三部分，字段完整；
3. JD 内容可编辑并保存版本；
4. 生成结果附带依据说明（如参考的岗位画像要素）。

### AC-US-02（简历分析）
1. 支持 PDF/Word 简历导入并成功解析；
2. 对每份简历输出 0–100 匹配评分、优势分析、风险提示；
3. 评分附带评分依据（命中/缺失的技能项）；
4. 批量导入后可按评分排序展示。

### AC-US-03（面试助手）
1. 录入面试记录后，系统生成面试总结、能力评价、推荐建议；
2. 推荐建议为「推荐/待定/不推荐」之一，并附依据；
3. 评价基于统一能力维度模板。

### AC-US-04（数据分析）
1. 看板展示招聘周期、转化率、渠道效果 ≥ 3 项指标；
2. 查询响应 ≤ 5s；
3. 支持按时间区间筛选。

### AC-US-06（权限与审计）
1. 不同角色仅能访问授权功能；
2. 所有 AI 调用留存可审计日志（输入摘要、输出、时间、操作人）；
3. 简历等敏感数据加密存储。

## 九、MVP 范围

### MVP 目标
验证「AI 驱动招聘三段式提效」核心价值闭环：**JD 生成 → 简历筛选 → 面试评价**，辅以最小可用的权限、审计与合规能力。

### MVP 包含
1. **AI 职位助手**：输入岗位信息 → 生成 JD / 岗位画像 / 技能要求（F1.1–F1.4）。
2. **AI 简历分析助手**：简历导入 → 结构化解析 → 匹配评分 + 优势 + 风险（F2.1–F2.5）。
3. **AI 面试助手**：面试记录录入 → 总结 + 能力评价 + 推荐建议（F3.1–F3.4）。
4. **系统支撑（最小集）**：用户登录与角色权限（F5.1）、AI 调用审计日志（F5.4）、数据脱敏与隐私保护（F5.5）。

### MVP 不包含（明确延后）
- 招聘数据分析看板（F4.x）→ P1 阶段；
- 批量简历筛选排序、面试问题建议、状态流转、岗位库管理（F2.6/F3.5/F5.2/F5.3）→ P1 阶段；
- JD 模板库、候选人画像对比、语音转写、自定义报表（F1.6/F2.7/F3.6/F4.5）→ P2 远期。

### MVP 成功标准
- 完整跑通一次招聘场景：生成 JD → 导入简历并评分 → 录入面试记录并生成评价；
- 性能达标（JD ≤15s，简历 ≤20s）；
- 所有 AI 输出可解释、有审计日志；
- 敏感数据加密存储，具备删除/导出能力。

## 十、待确认事项与假设

1. **技术栈**：项目 `project.md` 中前端/后端/数据库/部署技术栈仍为占位符，需与 Architect Agent 对齐后确认（PRD 不预设具体技术选型）。
2. **AI 能力来源**：JD 生成、简历解析、面试总结依赖 LLM 与可能的简历解析模型，具体模型与接入方式由架构阶段确定。
3. **数据来源**：招聘数据资产所需的历史数据是否就绪、历史简历可否导入，需业务侧确认。
4. **合规边界**：个人信息留存期限、删除策略需法务/合规确认后细化。
5. **优先级基准**：本版优先级以业务需求文档目标为依据，MVP 排除数据分析看板以确保核心闭环最快可验证；如业务侧要求看板同步上线，可上调 F4 部分项至 P0。
